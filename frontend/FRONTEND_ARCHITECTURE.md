# FRONTEND_ARCHITECTURE.md

# ReguTwin Agentic OS Frontend Architecture

## Purpose

This document explains the internal architecture of the ReguTwin frontend application.

The goal is to provide a clear understanding of:

* System structure
* Data flow
* Communication patterns
* State management
* API integration
* Realtime updates
* User interaction lifecycle

This document should be read before implementing any frontend feature.

---

# High-Level Architecture

The frontend acts as the presentation and interaction layer of the ReguTwin ecosystem.

It does not perform compliance analysis or AI processing.

Its responsibility is to visualize information generated by the backend and AI systems.

Architecture:

```txt
Users
  │
  ▼
Frontend (React)
  │
  ▼
Backend APIs
  │
  ├── MongoDB
  │
  ├── AI Layer
  │     ├── Watchman Agent
  │     ├── Analyst Agent
  │     ├── Mapper Agent
  │     ├── Validator Agent
  │     ├── LangGraph
  │     └── ChromaDB
  │
  ▼
Results Returned To Frontend
```

The frontend never communicates directly with AI services.

All requests must pass through backend APIs.

---

# Frontend Responsibilities

The frontend is responsible for:

* Authentication
* Authorization
* Dashboard visualization
* Regulation management
* MAP tracking
* Validation monitoring
* Conflict investigation
* Audit visibility
* Notification management
* Report generation

The frontend is not responsible for:

* LLM execution
* Embedding generation
* Vector search
* Regulation extraction
* Validation execution

These belong to the backend and AI layers.

The frontend should be divided into multiple layers.

```txt
UI Layer

     ▼
Feature Layer
     │
     ▼
State Layer
     │
     ▼
```

# UI Layer

The UI Layer contains all visual elements.

Examples:

* Buttons
* Inputs
* Cards
* Tables
* Modals
* Charts
* Layouts

Folder:

```txt
src/components/ui
```

The UI layer must remain reusable and independent from business logic.

---

# Feature Layer

The Feature Layer contains domain-specific functionality.

Examples:

```txt
Regulation Components

MAP Components

Validation Components

Audit Components

Dashboard Components
```

Folder:

```txt
src/components
```

Business-specific UI belongs here.

---

# State Layer

The State Layer manages application state.

Two types of state exist:

## Global State

Managed using Zustand.

Examples:

```txt
Authenticated User

Permissions

Theme

Notifications

Sidebar State
```

Store Location:

```txt
src/store
```

---

## Server State

Managed using React Query.

Examples:

```txt
Regulations

MAPs

Validation Results

Audit Logs

Conflict Reports
```

These values come from APIs.

Never store server state inside Zustand.

---

# API Layer

The API Layer is responsible for all backend communication.

Folder:

```txt
src/services
```

Example:

```txt
src/services/regulationService.ts

src/services/mapService.ts

src/services/validationService.ts

src/services/auditService.ts
```

The API layer should be the only place where Axios is used.

Components should never directly call Axios.

---

# Routing Architecture

Routes should be feature-based.

Example:

```txt
/
├── login
├── dashboard
├── regulations
├── regulations/:id
├── maps
├── maps/:id
├── conflicts
├── conflicts/:id
├── validation
├── validation/:id
├── audit
├── settings
```

Each feature should have its own page.

---

# Layout Architecture

The application should use shared layouts.

```txt
Auth Layout

Dashboard Layout

Admin Layout
```

Example:

```txt
src/layouts
```

The sidebar, navbar, and footer should exist only once inside layouts.

Pages should never duplicate navigation structures.

---

# Data Flow Architecture

Example:

A new RBI regulation arrives.

Workflow:

1. Watchman Agent detects regulation.

2. Analyst Agent extracts obligations.

3. Mapper Agent generates MAPs.

4. Backend stores data.

5. Backend API exposes results.

6. React Query fetches results.

7. Dashboard updates.

8. User views regulation.

Frontend never knows how AI produced the result.

Frontend only receives structured data.

---

# Dashboard Architecture

Dashboard widgets should be independent.

Example:

```txt
Compliance Score Card

Active Regulations Card

Pending MAPs Card

Conflict Alerts Card

Validation Status Card
```

Each widget should fetch its own data.

This improves scalability and maintainability.

---

# Real-Time Architecture

ReguTwin is intended to operate in near real time.

Socket.IO should be used.

Events:

```txt
new-regulation

map-created

validation-started

validation-failed

validation-passed

conflict-detected

notification-created
```

When an event arrives:

```txt
Socket Event
     │
     ▼
React Query Cache Update
     │
     ▼
UI Refresh
```

Avoid manual page refreshes.

---

# Authentication Architecture

Authentication flow:

```txt
Login

▼

Backend Verifies Credentials

▼

JWT Returned

▼

Token Stored

▼

Protected Routes Enabled
```

Protected routes should validate:

* Authentication
* User Role
* User Permissions

before rendering pages.

---

# Role-Based Access Control

Roles:

```txt
Compliance Officer

Auditor

Risk Team

IT Security

Administrator
```

Navigation should dynamically change according to permissions.

Example:

An Auditor should not access User Management.

---

# Notification Architecture

Notifications should be centralized.

Notification Sources:

```txt
New Regulation

Validation Failure

Conflict Detected

Deadline Approaching

Task Assigned
```

Storage:

```txt
notificationStore
```

Delivery:

```txt
REST API
+
Socket.IO
```

---

# Error Handling Architecture

Every API call should support:

```txt
Loading State

Success State

Error State

Retry State
```

Example:

```txt
Loading Regulations...

Failed to Load Regulations

Retry
```

Never leave blank screens.

---

# Performance Strategy

To keep the platform responsive:

Use:

* React Query caching
* Pagination
* Lazy loading
* Code splitting
* Route-based chunking

Avoid:

* Large initial bundles
* Duplicate API requests
* Excessive re-renders

---

# Security Principles

Never expose:

* OpenAI API Keys
* Gemini API Keys
* ChromaDB Credentials
* Internal Validation APIs

All sensitive operations must occur server-side.

The frontend should only consume secured backend endpoints.

---

# Frontend Design Principles

Every screen should answer one of these questions:

What regulations affect us?

What actions must be completed?

What conflicts exist?

What validations failed?

What evidence exists?

If a screen does not help answer one of these questions, it likely does not belong in the MVP.

---

# Architecture Goal

The frontend should function as the Compliance Command Center of ReguTwin.

A compliance officer should be able to:

* Understand new regulations
* View AI-generated obligations
* Track compliance tasks
* Detect conflicts
* Review validation evidence
* Generate audit reports

without interacting directly with the underlying AI infrastructure.

The frontend must make regulatory intelligence simple, transparent, actionable, and trustworthy.
