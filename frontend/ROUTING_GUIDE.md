# ROUTING_GUIDE.md

# ReguTwin Agentic OS Routing Guide

## Purpose

This document defines the complete routing architecture for the ReguTwin frontend.

It explains:

* Route hierarchy
* Navigation structure
* Layout usage
* Protected routes
* Role-based access control
* Future route expansion

The goal is to maintain a scalable and predictable routing system as the application grows.

---

# Routing Philosophy

Routes should be organized around business domains rather than UI components.

Bad Example:

```txt
/dashboard-page
/user-page
/table-page
```

Good Example:

```txt
/regs
/maps
/conflicts
/validation
/audit
```

Every route should represent a real compliance workflow.

---

# Route Protection Strategy

Every protected route should pass through:

```txt
Authentication Check

↓

Role Check

↓

Permission Check

↓

Page Render
```

Example:

```txt
User Requests Route

↓

Protected Route

↓

Verify JWT

↓

Verify Role

↓

Allow Access
```

If validation fails:

```txt
Redirect → /login
```

or

```txt
403 Unauthorized
```

```txt
│

│
├── conflicts
│   └── :id
│
├── validation
│   └── :id
│
├── audit
│
├── notifications
│
├── settings
│
└── admin
    ├── users
    ├── departments
    └── ai
```

This routing structure ensures that every major capability of ReguTwin—regulation intelligence, compliance actions, conflict detection, validation, and governance—has a dedicated navigation path while remaining scalable for future enterprise features.
